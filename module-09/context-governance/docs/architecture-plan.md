# 架构方案草案

团队计划先以单体架构交付第一个版本，核心目标是缩短上线周期、减少早期服务治理成本，并让业务团队能在一个代码库中快速迭代。系统包含用户、订单、账单、通知和运营后台五个模块。当前团队规模为 8 人，后端 4 人，前端 2 人，测试 1 人，产品 1 人。上线前三个月预计日活 2 万，峰值 QPS 约 300。

## 当前设计

代码采用分层结构：接口层、应用服务层、领域层和基础设施层。所有模块共享一个 PostgreSQL 数据库，模块之间通过 Go interface 调用。发布方式为单镜像部署，使用蓝绿发布和数据库迁移脚本。日志、指标和链路追踪在第一个版本接入，故障恢复目标为 30 分钟内恢复核心下单链路。

## 已确认约束

1. 第一个版本需要在 8 周内交付，当前团队没有独立平台工程能力。
2. 订单和账单存在强一致性要求，短期内不希望引入分布式事务。
3. 运营后台变化频繁，可能影响用户和订单模块的模型边界。
4. 数据库表会由多个模块访问，若缺少所有权约束，后续拆分成本会上升。
5. 通知模块可以较早拆出，因为它和核心交易链路耦合较低。

## 主要风险

数据层风险最高。共享数据库会让模块边界变得模糊，多个模块直接读写同一张表时，未来拆分服务需要梳理数据所有权、迁移历史数据、补偿双写窗口，并处理跨服务查询。若早期没有建立表级所有者和访问规则，后期拆分会变成高风险数据库工程。

模块依赖风险也需要关注。单体代码库容易让开发者跨层调用内部实现，短期看能加快功能交付，长期会形成隐式耦合。建议用包边界、接口契约、架构测试和代码评审规则约束依赖方向。

发布风险来自单体发布粒度。任何模块变更都需要整体发布，测试范围会随着模块增加而扩大。若运营后台频繁变化，可能拖累核心交易链路发布节奏。建议为高频变化模块建立更强测试隔离，并把灰度和回滚脚本作为必需交付物。

组织风险来自团队增长。当前 8 人团队可以使用单体协作；当后端超过 10 人、同一模块多人并行改动、发布冲突明显增加时，需要评估拆分。拆分触发条件应写入架构决策记录。

## 拆分触发指标

- 单次构建和测试超过 15 分钟；
- 每周出现 3 次以上模块间变更冲突；
- 核心交易链路和运营后台发布节奏明显冲突；
- 单次发布回滚影响多个无关模块；
- 某个模块需要独立扩容或独立合规审计；
- 数据表访问路径无法清晰归属到单一模块。

## 建议行动

第一阶段保留单体，重点治理模块边界。每个模块必须声明数据所有权、对外接口和禁止访问的内部包。第二阶段建立架构守护测试，自动检测跨模块非法依赖。第三阶段为通知、运营后台等低耦合模块准备拆分预案。第四阶段每两周复盘一次拆分触发指标，指标满足时进入拆分评审。

## 汇报摘要

单体起步可以降低早期复杂度，但需要把未来拆分的约束前置。关键做法是在第一天就让模块边界、数据所有权、发布回滚和监控指标可见。只要边界清楚、指标透明、回滚可靠，单体可以成为阶段性策略；若边界失控，单体会积累成难以拆解的技术债。
